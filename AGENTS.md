# AGENTS.md — o7am.github.io

This document describes how the o7AM website is built, deployed, and structured so that agents and developers can work on it effectively.

## Overview

The site is a **static HTML site** generated by a Python script. Content and copy live in YAML under `data/`. Jinja2 templates under `templates/` are rendered once per locale (Polish `pl` and English `en`). The output is written to `out/` and deployed to **GitHub Pages** via a GitHub Actions workflow. The published site is served from the contents of `out/` (not from the repo root).

- **Site URL:** https://o7.am  
- **Locales:** Polish (root), English (`en/`)  
- **Build:** `build.py` → writes to `out/`

---

## Build process

### Local build

- **Recommended:** `just build` (uses [Just](https://github.com/casey/just) and runs `uv run python build.py`).
- **Alternatively:** `uv run python build.py` or `python build.py` with dependencies installed via `pip install -r requirements.txt`.

The script:

1. Loads YAML from `data/` (i18n, blogs, portfolios).
2. For each locale (`pl`, `en`), generates a list of pages (output path, template name, context).
3. Renders each page with Jinja2, writing HTML into `out/` (e.g. `index.html`, `en/index.html`, `blog.html`, `portfolios/3d_prints.html`, etc.).
4. Copies static files and dirs from the repo root into `out/` (see [Static files](#static-files)).
5. Overwrites specific generated portfolio pages with hand-written HTML (see [Static portfolio overrides](#static-portfolio-overrides)).
6. Writes `out/en.html` as a redirect to `en/index.html` (for `/en`-style URLs on GitHub Pages).

**Output directory:** `out/`. It is in `.gitignore`; do not commit it. The CI workflow uploads `out/` as the GitHub Pages artifact.

### Preview locally

- `just serve` — serves `out/` at http://localhost:8000.

### Deployment (CI)

- **Workflow:** `.github/workflows/deploy-pages.yml`
- **Trigger:** Push to `main`
- **Steps:** Checkout → `pip install -r requirements.txt` → `python build.py` → upload `out/` as `upload-pages-artifact` → `deploy-pages` deploys that artifact to GitHub Pages.
- **Pages source:** In repo **Settings → Pages**, the source must be **GitHub Actions** (not “Deploy from a branch”).

## File and directory reference

### Root

| File / dir       | Purpose |
|-----------------|--------|
| `build.py`      | Static site generator. Loads `data/*.yaml`, renders `templates/`, writes to `out/`, copies static assets and applies overrides. |
| `justfile`      | Commands: `just build` (build), `just serve` (preview `out/` on port 8000). Uses `uv run` for the build. |
| `requirements.txt` | Python deps for the build: `Jinja2`, `PyYAML`. Used by both local `uv`/`pip` and CI. |
| `style.css`      | Global styles. Copied into `out/` as-is. |
| `app.js`        | Global front-end script (e.g. menu, lazy loading). Copied into `out/`. |
| `llms.txt`      | [llms.txt](https://txt-llms.com/documentation) manifest for LLM search engines. Markdown summary and links; copied to `out/`. |
| `robots.txt`     | Crawler rules. Copied to `out/`. |
| `sitemap.xml`    | Sitemap for search engines. Copied to `out/`. |
| `CNAME`          | Custom domain for GitHub Pages (e.g. `o7.am`). Copied to `out/`. |
| `.gitignore`     | Ignores `out/`, `.venv/`, `.worktrees/`, `.docs/`, macOS junk. |
| `AGENTS.md`      | This file. |

### `data/`

All content and copy used by the build. Keys are referenced in templates (e.g. `t.nav.services`, `item.title[locale]`).

| File | Purpose |
|------|--------|
| `data/i18n.yaml` | Per-locale strings: nav labels, meta titles/descriptions, services headings, footer, contact. Structure: `pl` / `en` → `nav`, `meta`, `services`, `blog`, `portfolio`, `contact`, `footer`. |
| `data/blogs.yaml` | List of blog posts. Each: `id`, `slug` (filename), `title`/`description`/`body`/`subtitle` per locale, `date`, `image`. Rendered with `blog_post.html`. |
| `data/portfolios.yaml` | List of portfolio items. Each: `id`, `slug`, `title`/`image`/`body` per locale, optional `gallery`. Rendered with `portfolio_item.html` unless overridden (see below). |

### `templates/`

Jinja2 templates. The build passes a shared context: `base` (path to **locale root** — use for same-locale page links so nav stays in the current language), `root` (path to **site root** — use for static assets: CSS, JS, `assets/`), `locale`, `t` (i18n for current locale), `site_url`, `canonical_url`, `alternate_*`, `other_lang_href`, plus page-specific vars.

| Template | Purpose |
|----------|--------|
| `base.html` | Layout: `partials/head.html`, header, `{% block main %}`, CTA, footer; loads `app.js`. |
| `partials/head.html` | Meta, title, description, Open Graph, Twitter, theme, CSS (`root`), canonical, hreflang. |
| `partials/header.html` | Logo, nav (services submenu, portfolio, blog), language switcher, contact button. Uses `base` for in-locale links, `t.nav.*`, `other_lang_href`. |
| `partials/footer.html` | Footer content and links. Includes Ko-fi support link in the socials row (first icon); same footer is injected into static portfolio pages at build time. |
| `partials/cta.html` | CTA block (e.g. “o7AM” branding). |
| `index.html` | Homepage. |
| `blog.html` | Blog listing; receives `blogs`. |
| `blog_post.html` | Single post; receives `blog`, `title`, `description`. |
| `portfolio.html` | Portfolio listing; receives `portfolios`. |
| `portfolio_item.html` | Single portfolio item: title, image, optional body, optional gallery; receives `item`. |
| `services.html` | Services overview. |
| `service_3d.html` | 3D printing service page. |
| `service_de.html` | Data Engineering service page. |
| `contact.html` | Contact page. |

### `assets/`

Images and other static assets. Copied as a whole into `out/assets/` (replace). Paths in data/templates are relative to site root (e.g. `assets/portfolios/1/cover.jpg`).

### `portfolios/` and `en/portfolios/` (source only)

Hand-written HTML that **replaces** some generated portfolio pages:

| File | Purpose |
|------|--------|
| `portfolios/emoji.html` | Custom emoji/kaomoji grid page (PL). Replaces the generic `portfolio_item.html` output for slug `emoji`. |
| `en/portfolios/emoji.html` | Same page in English. |

The build still generates `portfolios/emoji.html` and `en/portfolios/emoji.html` from data, then overwrites them with these files if present. Other portfolio slugs (e.g. `3d_prints`) are only generated from `portfolios.yaml` and `portfolio_item.html`.

### `scripts/`

| Script | Purpose |
|--------|--------|
| `scripts/sync-en-from-pl.py` | **PL/EN sync check.** Compares Polish vs English content in `data/i18n.yaml`, `data/blogs.yaml`, and `data/portfolios.yaml`. Reports: keys present in PL but missing in EN; EN values identical to PL (likely untranslated). Run: `just sync-en` or `uv run python scripts/sync-en-from-pl.py`. Use `--format json` for machine output; use `--suggest` to print suggested EN snippets (copy from PL) for manual translation. See [PL/EN sync command](#plen-sync-command) below. |

### `docs/`

Design and planning docs (e.g. `docs/plans/`). Not used by the build; for humans/agents.

### `.github/workflows/`

| File | Purpose |
|------|--------|
| `deploy-pages.yml` | Builds the site with Python, uploads `out/` as the Pages artifact, deploys to GitHub Pages. |

### Generated in `out/` (do not edit by hand)

- `out/en.html` — Redirect to `en/index.html` (so `/en.html` works on GitHub Pages).
- All HTML under `out/` and `out/en/` is produced by `build.py` (except where overwritten by static portfolio files above).

## Adding or changing content

- **Copy / UI strings:** Edit `data/i18n.yaml` (per locale).
- **Blog posts:** Add or edit entries in `data/blogs.yaml`; keep `slug` stable (used in URLs and filenames).
- **Portfolio items:** Add or edit entries in `data/portfolios.yaml`. Use `body.pl` / `body.en` for HTML description. To use a **custom page** instead of the generic template, add the slug to `STATIC_PORTFOLIOS` in `build.py` and add the corresponding `portfolios/<slug>.html` and `en/portfolios/<slug>.html` files.
- **New pages (e.g. new service):** Add a new template, then in `build.py` extend the `pages` list (per locale) with the correct output path, template name, and context.

### Theme-aware styling (light/dark)

The site supports a light/dark theme (`data-theme="light"` or `"dark"` on `<html>`). **Any new or changed text and UI must be theme-oriented:** use CSS variables (e.g. `var(--text-primary)`, `var(--bg-page)`, `var(--header-text)`) instead of hardcoded colors like `#fff` or `#000`. That way new copy, borders, and backgrounds respect the user’s theme. See `style.css` for the existing theme variables and `[data-theme="light"]` overrides.

### Support link (Ko-fi)

A “support us” / “buy me a coffee” link points to Ko-fi (`https://ko-fi.com/o7am_`). It appears in the **footer** as the first social icon (same style as Instagram/LinkedIn) and on the **contact page** as a green round button with a coffee icon. Copy: `footer.support_us`, `contact.support_line`, `contact.support_cta` in `data/i18n.yaml`. No backend; link-only.

### PL/EN sync command

To check whether English content is in sync with Polish (e.g. after adding new copy in PL):

- **Run:** `just sync-en` or `uv run python scripts/sync-en-from-pl.py` (from repo root).
- **Output:** A report listing (1) keys present in PL but missing in EN (`data/i18n.yaml`), and (2) EN values that are identical to PL (often intentional e.g. "Portfolio", "Data Engineering"; review and translate if not).
- **Options:** `--format json` for machine-readable output; `--suggest` to print suggested EN snippets (copy from PL) for missing or identical i18n keys, for manual translation.
- **When to run:** After editing `data/i18n.yaml`, `data/blogs.yaml`, or `data/portfolios.yaml`; before committing large content changes; when preparing a release.

- **STATIC_FILES:** `style.css`, `app.js`, `CNAME`, `robots.txt`, `sitemap.xml`, `llms.txt`

## Static portfolio overrides

Slugs listed in `STATIC_PORTFOLIOS` in `build.py` (e.g. `emoji`) get their generated `portfolios/<slug>.html` and `en/portfolios/<slug>.html` overwritten by the files in the repo under `portfolios/` and `en/portfolios/` if those files exist. Use this for pages that need custom layout or script (e.g. the emoji grid) instead of the generic portfolio template. The build **injects the shared footer partial** (from `templates/partials/footer.html`) into these static files by replacing `<!-- FOOTER_PARTIAL -->` with the rendered footer, so the Ko-fi link and other footer content stay in sync without duplicating markup.

## Locales and URLs

- **Polish (default):** Root paths, e.g. `/`, `/portfolio.html`, `/portfolios/3d_prints.html`.
- **English:** Under `en/`, e.g. `/en/index.html`, `/en/portfolio.html`, `/en/portfolios/emoji.html`.
- **Language switcher:** Uses `other_lang_href` (e.g. `en/index.html` from PL index). For the EN homepage, use `https://o7.am/en/index.html` or `https://o7.am/en.html` (redirect).
- **Nav persistence:** Header and footer links use `base` (locale root), so clicking e.g. Contact from the English site goes to `/en/contact.html` and keeps the chosen language.

## Summary for agents

1. **Build:** Run `just build` or `uv run python build.py`; output is in `out/`.
2. **Content:** Edit YAML in `data/`; templates in `templates/` consume it.
3. **Deploy:** Push to `main`; the workflow builds and deploys `out/` to GitHub Pages.
4. **Custom portfolio pages:** Add HTML under `portfolios/` and `en/portfolios/`, and add the slug to `STATIC_PORTFOLIOS` in `build.py`. Use `<!-- FOOTER_PARTIAL -->` where the footer should go; the build injects the shared footer (including Ko-fi) there.
5. **New static assets:** Put them in `assets/` or add to `STATIC_FILES` if at root; they will be copied into `out/` on build.
6. **Theme-aware UI:** Any new or changed text/UI must use CSS variables (e.g. `var(--text-primary)`) so it works in both light and dark theme; avoid hardcoded colors.
7. **Support link:** Ko-fi lives in the footer (first social icon) and on the contact page (green coffee button); copy in i18n (`footer.support_us`, `contact.support_*`).
8. **PL/EN sync:** After editing `data/i18n.yaml`, `data/blogs.yaml`, or `data/portfolios.yaml`, run `just sync-en` (or `uv run python scripts/sync-en-from-pl.py`) to see if EN is missing keys or still has copy identical to PL (needs translation).
